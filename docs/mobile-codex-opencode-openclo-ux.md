# 오픈코드/오픈클로 사용성을 모바일 GPT 앱 Codex 기능으로 재현하기

## 1) 목표 정의

모바일 GPT 앱 안의 **Codex 기능**만으로, 사용자가 오픈코드/오픈클로에서 느끼는 핵심 경험을 최대한 비슷하게 재현한다.

- **빠른 실행감**: “요청 → 코드 생성/수정 → 결과 확인” 루프를 짧게 유지
- **낮은 진입장벽**: 복잡한 설정 없이 템플릿 기반으로 시작
- **작업 연속성**: 대화 맥락과 작업 히스토리를 쉽게 이어가기
- **가시성**: 변경점, 실패 원인, 다음 액션이 명확히 보이기

---

## 2) 재현해야 할 핵심 사용성(UX) 요소

### A. 원탭 시작(Quick Start)

오픈코드/오픈클로의 장점은 “무엇을 해야 할지”보다 “바로 시작”에 있다.

모바일 Codex에서 아래 3가지 고정 진입 버튼을 제안한다.

1. **버그 고치기**
2. **기능 추가**
3. **리팩터링**

각 버튼은 내부적으로 프롬프트 템플릿을 자동 삽입한다.

- 버그 고치기 템플릿 예시
  - 현재 증상
  - 기대 동작
  - 재현 단계
  - 관련 파일/로그

### B. 구조화 입력(폼 + 자연어 혼합)

모바일에서는 긴 프롬프트 타이핑이 불편하므로, 다음 혼합 구조를 사용한다.

- 상단: 짧은 자유 입력(한 줄 지시)
- 하단: 선택형 필드
  - 언어/프레임워크
  - 우선순위(속도/안정성/가독성)
  - 변경 범위(작게/중간/크게)

Codex에는 이 정보를 system/developer 컨텍스트 형태로 전달해, 사용자가 매번 장문을 입력하지 않아도 안정된 결과를 얻도록 한다.

### C. 변경점 우선 표시(Diff-First)

모바일 코드뷰는 한 화면에서 맥락을 파악하기 어렵다. 따라서 결과 화면의 기본값은 “설명”이 아니라 “변경 파일 + 핵심 diff”로 둔다.

권장 카드 순서:

1. 수정 파일 목록
2. 각 파일 핵심 변경 요약(3줄)
3. 전체 패치 보기
4. 테스트 결과
5. 후속 제안

### D. 실패 복구 UX(Recovery by Default)

오픈코드/오픈클로에서 만족도가 높은 지점은 실패해도 다음 단계가 명확한 점이다.

모바일 Codex도 실패 시 자동으로 다음을 제시한다.

- 실패 원인 1문장 요약
- 가장 가능성 높은 수정안 2개
- “이대로 다시 시도” 버튼
- “범위 줄여서 시도” 버튼

### E. 작업 히스토리 스냅샷

세션 단위로 아래 항목을 자동 저장한다.

- 요청 요약
- 변경 파일
- 실행한 테스트/결과
- 롤백 포인트

사용자는 이전 스냅샷을 선택해 “여기서 이어서”를 바로 실행할 수 있다.

---

## 3) 모바일 Codex 정보 구조(IA) 제안

하단 탭 4개 구조:

1. **시작**: Quick Start 버튼 3종 + 최근 작업
2. **작업**: 현재 세션의 대화/패치/테스트
3. **히스토리**: 스냅샷 리스트 + 비교
4. **설정**: 기본 프롬프트, 코드 스타일, 테스트 기본값

핵심은 “설정은 늦게, 실행은 먼저” 원칙이다.

---

## 4) 사용자 플로우(대표 2개)

### 플로우 1: 버그 수정

1. 시작 탭 → `버그 고치기`
2. 증상 한 줄 입력 + 로그 첨부
3. Codex가 원인 가설/수정안 생성
4. Diff-First 카드로 변경점 검토
5. 테스트 실행 결과 확인
6. 적용 또는 롤백

### 플로우 2: 기능 추가

1. 시작 탭 → `기능 추가`
2. 목표/제약(성능, 호환성) 입력
3. Codex가 구현안 2개 제시(보수/공격)
4. 사용자가 옵션 선택
5. 패치 + 테스트 + 사용 예시 생성
6. 스냅샷 저장 후 다음 작업으로 이동

---

## 5) 프롬프트 템플릿(모바일 최적화)

### 템플릿: 버그 고치기

```
목표: 아래 버그를 최소 수정으로 해결.
증상: {symptom}
기대 동작: {expected}
재현 단계: {steps}
제약: 공개 API 변경 금지, 테스트 추가 필수.
출력 형식:
1) 원인 요약(최대 3개)
2) 권장 수정안 1개
3) 파일별 diff
4) 테스트/검증 명령
5) 롤백 방법
```

### 템플릿: 기능 추가

```
목표: 아래 기능을 기존 구조를 존중하며 추가.
요구사항: {requirements}
비기능 요구: {nfr}
제약: 의존성 추가 최소화, 하위호환 유지.
출력 형식:
1) 구현 전략 2안
2) 선택안 근거
3) 파일별 변경 계획
4) 패치
5) 테스트/문서 업데이트
```

## 6) 성공 지표(KPI)

출시 후 최소 2주 단위로 측정:

- **TTV(Time to Value)**: 첫 유효 결과까지 걸린 시간
- **FCR(First Completion Rate)**: 첫 시도에서 사용자가 “적용”까지 간 비율
- **Retry Burden**: 재시도 평균 횟수
- **Diff Comprehension Rate**: 사용자가 diff 이해에 성공한 비율(간단 설문)

목표 예시:

- TTV 40% 단축
- FCR 20% 향상
- Retry Burden 30% 감소

---

## 7) 단계별 구현 로드맵

### Phase 1 (1~2주)
- Quick Start 3버튼
- 템플릿 프롬프트 적용
- Diff-First 결과 카드

### Phase 2 (2~4주)
- 실패 복구 버튼 2종
- 스냅샷 저장/복원
- 기본 KPI 수집 대시보드

### Phase 3 (4주+)
- 사용자별 프롬프트 개인화
- 작업 유형별 추천 자동화
- 팀 공유(스냅샷/패치 링크)

---

## 8) 구현 체크리스트

- [ ] 모바일 입력 폼(자유 입력 + 구조화 필드) 반영
- [ ] 결과 화면 기본 탭을 diff로 설정
- [ ] 실패 시 자동 대안 2개 생성 로직 연결
- [ ] 세션 스냅샷 저장 스키마 정의
- [ ] KPI 이벤트 로깅 포인트 확정
- [ ] 롤아웃 A/B 실험 설계

## 9) 한 줄 결론

모바일 Codex에서 오픈코드/오픈클로 사용성을 재현하려면, 모델 성능 경쟁보다 **입력 단순화 + 변경점 가시화 + 실패 복구 자동화**의 UX 3축을 우선 구현하는 것이 가장 효과적이다.
